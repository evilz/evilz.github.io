Dans cette vidéo, Bumble dev revient sur le principe de responsabilité unique et la manière de découper une classe ou un module autour d'une seule raison de changer. L'objectif est de montrer comment ce principe aide à écrire un code plus lisible, plus testable et surtout plus simple à faire évoluer dans le temps.

Le format reste très concret : on part d'un problème de conception courant, puis on voit comment le principe SOLID permet de mieux découper les responsabilités et d'éviter des dépendances inutiles. C'est une bonne ressource pour revoir les fondamentaux de l'architecture objet avec des exemples accessibles.
